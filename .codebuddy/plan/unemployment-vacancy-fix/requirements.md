# 需求文档：修复 Tier 0/1 空缺岗位失业人口不就业问题

## 引言

当前就业系统中存在一个 bug：当 Tier 0/1 岗位（peasant、lumberjack、miner、worker、serf）存在空缺时，失业人口未能及时填补这些岗位。Tier 0/1 岗位的填补路径是"直接雇佣"（无财富门槛），理论上失业者应该能立即填入，但实际游戏中出现了持续性的失业与空缺并存现象。

### 根因分析

通过代码审查，确认以下因素共同导致了该问题：

1. **填补速率限制过低**：`VACANCY_FILL_RATIO_PER_TICK = 0.25`，每个角色每 tick 最多只能填补 25% 的空缺。如果有 100 个空缺，需要 4+ tick 才能填满。在游戏早期或建筑快速扩张时，空缺增长速度可能超过填补速度。

2. **Tier 0/1 岗位之间的竞争消耗**：`vacancyRanking` 按净收入从高到低排序，多个 Tier 0/1 岗位依次从同一个失业池中取人。高收入的 Tier 0/1 岗位先消耗失业者，低收入的 Tier 0/1 岗位可能分不到人。

3. **净收入估算不准确**：`getSmartExpectedWage` 对低人口岗位使用潜力预估，但对已有一定人口的岗位使用历史工资。如果某个 Tier 0/1 岗位的历史工资被低估（例如刚解锁的建筑还没有产出记录），其净收入排名会偏低，导致填补优先级下降。

4. **每 tick 只执行一次 fillVacancies**：不像 handleJobMigration 那样有多轮迭代机会，fillVacancies 每 tick 只跑一次，限制了填补效率。

## 需求

### 需求 1：提高 Tier 0/1 岗位的填补速率

**用户故事：** 作为一名玩家，我希望失业人口能更快地填补低级岗位空缺，以便建筑能尽快投入生产。

#### 验收标准

1. WHEN Tier 0/1 岗位存在空缺 AND 失业池中有可用人口 THEN 系统 SHALL 以不低于 50% 的速率（每 tick）填补 Tier 0/1 空缺（即 Tier 0/1 使用独立的、更高的填补速率常量）。
2. WHEN 失业人口数量大于所有 Tier 0/1 空缺总和 THEN 系统 SHALL 在 2 tick 内将所有 Tier 0/1 空缺填满。
3. WHEN Tier 2/3 岗位的填补速率保持不变（仍为 25%）THEN 系统 SHALL 不影响高级岗位的社会流动平衡。

### 需求 2：优先保障 Tier 0/1 岗位的填补顺序

**用户故事：** 作为一名玩家，我希望基础生产岗位（农民、矿工、工人等）优先于中高级岗位被填补，以便经济基础不会因为就业分配顺序而受损。

#### 验收标准

1. WHEN fillVacancies 处理空缺列表时 THEN 系统 SHALL 先处理所有 Tier 0/1 空缺，再处理 Tier 2/3 空缺（在同一 Tier 内仍按净收入排序）。
2. IF Tier 0/1 空缺总数超过失业人口 THEN 系统 SHALL 按净收入从高到低在 Tier 0/1 岗位之间分配可用失业者。
3. WHEN Tier 0/1 空缺被优先处理后 THEN 系统 SHALL 将剩余失业者（如有）用于 Tier 2/3 岗位的候选人池。

### 需求 3：改善新解锁岗位的工资预估

**用户故事：** 作为一名玩家，我希望新建建筑提供的岗位能吸引到失业者，以便新建筑不会因为工资预估过低而长期空置。

#### 验收标准

1. WHEN 某个 Tier 0/1 角色的当前人口为 0 或低于 LOW_POP_THRESHOLD THEN 系统 SHALL 使用 `getSmartExpectedWage`（取潜力预估和历史工资的较大值）来计算该角色的净收入排名。
2. WHEN 某个 Tier 0/1 角色的净收入估算值为 0 或负数 THEN 系统 SHALL 使用一个最低保底值（如平均工资的 50%）来确保该角色不会被排到最后。
3. IF 所有 Tier 0/1 岗位的净收入均为正 THEN 系统 SHALL 按实际净收入排序，不施加额外干预。

### 需求 4：确保数值一致性和无副作用

**用户故事：** 作为一名开发者，我希望修复不会引入新的经济失衡或人口振荡问题，以便游戏的整体平衡不受影响。

#### 验收标准

1. WHEN 修改填补速率和排序逻辑后 THEN 系统 SHALL 不改变 Tier 2/3 岗位的填补机制（财富门槛、Lucky Promotion 等）。
2. WHEN 失业池为空时 THEN 系统 SHALL 不从已就业的角色中强制抽调人口来填补空缺（这是 handleJobMigration 的职责）。
3. WHEN 大量人口同时被裁员（如建筑被摧毁）THEN 系统 SHALL 在 2-3 tick 内将裁员人口重新分配到有空缺的 Tier 0/1 岗位，而非长期滞留在失业池。
4. WHEN 修改后的常量值发生变化 THEN 系统 SHALL 将新常量定义在 `src/logic/utils/constants.js` 中，遵循现有命名规范。
