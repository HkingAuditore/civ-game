# 需求文档：军团自动补兵系统修复

## 引言

当前游戏引入了军团系统（Corps System）后，原有的自动补兵机制出现了断裂。问题的核心在于：

1. **战损发生在军团内部**：战斗结束后，存活单位直接写回军团的 `corps.units`（通过 `distributeSurvivorsToCorps`），战损同时从全局 `army` 扣除。
2. **补兵只补到散兵池**：自动补兵逻辑（`AUTO_REPLENISH_LOSSES` 信号 → 训练队列 → 训练完成后加入 `army`）将新单位放入全局散兵池，而非回补到受损军团。
3. **结果**：玩家在战后必须手动将训练好的新单位重新编入军团，自动补兵形同虚设。

本需求旨在扩展现有自动补兵机制，使其能感知军团系统，将补充的单位自动回补到受损军团中，恢复自动补兵的实际作用。

### 现有机制概览

| 模块 | 文件 | 职责 |
|------|------|------|
| 军团系统 | `src/logic/diplomacy/corpsSystem.js` | 军团创建/解散、单位编入/撤出 |
| 战斗系统 | `src/logic/diplomacy/battleSystem.js` | 战斗计算、伤亡结算 |
| 游戏循环 | `src/hooks/useGameLoop.js` | 自动补兵信号接收、训练队列处理、训练完成入列 |
| 游戏动作 | `src/hooks/useGameActions.js` | `handleAutoReplenishLosses`（补兵逻辑） |
| 军团管理UI | `src/components/panels/CorpsManagementPanel.jsx` | 军团管理面板 |
| 军事Tab | `src/components/tabs/MilitaryTab.jsx` | 自动补兵开关UI |

### 数据流现状

```
战斗结束 → 存活单位写回 corps.units
         → 战损从 army 扣除
         → AUTO_REPLENISH_LOSSES 日志信号发出
         → 自动补兵逻辑：计算损失 → 扣资源 → 加入训练队列
         → 训练完成 → 新单位加入 army 散兵池
         → ❌ 断裂：新单位留在散兵池，不会自动回到军团
```

## 需求

### 需求 1：训练完成的补兵单位自动回补到受损军团

**用户故事：** 作为一名玩家，我希望自动补兵训练完成的单位能自动编入受损军团中，以便我不需要在每次战斗后手动将补充兵力重新分配到军团。

#### 验收标准

1. WHEN 标记为 `isAutoReplenish: true` 的训练队列项目完成训练 THEN 系统 SHALL 检查是否存在需要补兵的军团（即该兵种在军团中的数量低于战前水平）。
2. IF 存在一个或多个受损军团缺少该类型单位 THEN 系统 SHALL 将完成训练的单位自动编入缺少该类型单位最多的军团中（优先补满最缺兵的军团）。
3. IF 目标军团当前处于 `in_combat` 状态 THEN 系统 SHALL 将该单位放入散兵池（`army`），不打断战斗中的军团编制。
4. IF 没有任何军团缺少该类型单位 THEN 系统 SHALL 将该单位放入散兵池（`army`），保持当前行为。
5. WHEN 单位被自动编入军团 THEN 系统 SHALL 在日志中显示补兵目标军团信息（例如："[自动补兵] 长矛兵 已补充到 第1军团"）。

### 需求 2：记录军团战前编制快照用于补兵目标计算

**用户故事：** 作为一名玩家，我希望系统能记住军团在战斗前的兵种构成，以便自动补兵能准确知道每个军团缺少多少什么类型的单位。

#### 验收标准

1. WHEN 自动补兵信号（`AUTO_REPLENISH_LOSSES`）被触发 THEN 系统 SHALL 同时记录损失来源军团的 ID 和对应的各兵种损失数量。
2. IF 多个军团参与同一场战斗 THEN 系统 SHALL 按军团 ID 分别记录各自的损失明细。
3. WHEN 补兵完成且军团已恢复到战前编制水平 THEN 系统 SHALL 清除该军团的待补兵记录。
4. IF 军团被解散 THEN 系统 SHALL 清除该军团对应的待补兵记录，后续补兵的单位进入散兵池。

### 需求 3：战线摩擦造成的损失也纳入自动补兵

**用户故事：** 作为一名玩家，我希望不仅战斗结算的损失会触发自动补兵，战线摩擦（front friction）造成的军团损失也能触发自动补兵，以便军团在持续消耗战中保持战斗力。

#### 验收标准

1. WHEN 战线摩擦（`processFrontFriction`）导致玩家军团产生伤亡 THEN 系统 SHALL 将这些损失按军团 ID 记录为自动补兵目标。
2. WHEN 摩擦损失积累超过一定阈值（建议每军团累计 ≥ 3 人）THEN 系统 SHALL 触发一次自动补兵流程，避免每个 tick 的微小损失都触发独立的补兵操作。

### 需求 4：自动补兵设置支持军团级别的开关

**用户故事：** 作为一名玩家，我希望能为每个军团独立设置是否启用自动补兵，以便我可以选择性地为重要军团保持满编，而让次要军团自然消耗。

#### 验收标准

1. WHEN 玩家在军团管理面板查看某个军团时 THEN 系统 SHALL 显示一个"自动补兵"开关，默认跟随全局自动补兵设置。
2. IF 全局自动补兵关闭 THEN 所有军团的自动补兵均不生效，无论单个军团设置如何。
3. IF 全局自动补兵开启 AND 某军团的自动补兵被单独关闭 THEN 该军团的战损不触发自动补兵，训练完成的单位不会自动编入该军团。
4. WHEN 军团的自动补兵设置变更 THEN 系统 SHALL 在日志中记录变更（例如："第1军团 自动补兵已关闭"）。

### 需求 5：补兵优先级和资源分配策略

**用户故事：** 作为一名玩家，我希望当多个军团同时需要补兵时，系统能按合理的优先级分配有限的训练槽位和资源，以便前线军团优先得到补充。

#### 验收标准

1. WHEN 多个军团同时存在补兵需求 THEN 系统 SHALL 按以下优先级排序：
   - 已部署到活跃战线（`deployed`）的军团优先
   - 缺员比例更高的军团优先
   - 同等条件下，创建时间更早的军团优先
2. IF 训练槽位或资源不足以满足所有军团的补兵需求 THEN 系统 SHALL 优先满足高优先级军团的需求，低优先级军团排后。
3. WHEN 补兵因资源不足而部分满足时 THEN 系统 SHALL 在日志中提示（例如："[自动补兵] 资源不足，优先补充 第1军团"）。

## 边界情况和技术考虑

### 边界情况

1. **军团在补兵期间被解散**：补兵目标军团不存在时，训练中的单位应正常完成训练并进入散兵池。
2. **军团在补兵期间开始新战斗**：训练完成时军团处于 `in_combat` 状态，单位暂入散兵池，战斗结束后下一轮补兵循环会重新评估。
3. **散兵池中已有同类型单位**：自动补兵只处理标记为 `isAutoReplenish` 的训练完成项，不会从散兵池中现有单位中抽调。
4. **多次战斗间的损失累计**：同一军团在多次战斗中的损失应累加记录，补兵按累计缺额计算。
5. **玩家手动编入单位**：玩家手动将单位编入军团后，该军团的待补兵缺额应相应减少。

### 技术约束

1. 必须在现有 `useGameLoop.js` 和 `useGameActions.js` 的训练完成逻辑中扩展，不创建新的并行系统。
2. 军团待补兵记录可以作为一个新的状态字段（如 `corpsReplenishQueue`），存储在 `useGameState` 中。
3. 补兵分配逻辑应尽量轻量，避免在每个 tick 中进行复杂计算（可在训练完成时才触发分配逻辑）。
4. 战斗损失信号（`AUTO_REPLENISH_LOSSES`）需要扩展为包含军团 ID 信息。

### 成功标准

1. 战斗结束后，自动补兵训练完成的单位自动进入受损军团，无需玩家手动操作。
2. 战线摩擦的持续损耗也能被自动补兵覆盖。
3. 多军团场景下补兵分配合理，前线军团优先。
4. 不影响现有散兵模式（无军团时）的自动补兵行为。
5. 日志信息清晰，玩家能看到补兵去向。
