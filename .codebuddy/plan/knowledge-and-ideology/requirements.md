# 知识与理念系统 需求文档

## 引言

本文档定义了 civ-game 中两个紧密关联的核心系统的改造与新增：

1. **知识系统（原"科技"系统重构）**：将原有的平铺式科技列表改造为真正的树状结构（知识树），强制要求线性前置解锁，参考《文明6》科技树和《维多利亚3》的科技网络。原有的"科技" Tab 将改名为"知识"Tab，科技树作为其中一个子 Tab。

2. **理念系统（全新系统）**：引入一套具有深度构建策略的卡牌系统，参考《小丑牌 (Balatro)》的 build 思路和《杀戮尖塔 (Slay the Spire)》的卡牌获取/强化机制。玩家通过游戏行为积累"理念分数"，达到阈值时从随机涌现的理念中三选一。被选择的理念可以装备到有限的"卡槽"中，产生深刻影响游戏各系统的效果，包含简单数值加成和复杂的联动/条件触发效果。理念作为"知识" Tab 下的另一个子 Tab 呈现。

---

## 第一部分：知识系统（科技树重构）

### 需求 1.1 — 命名与概念重构

**用户故事：** 作为一名玩家，我希望游戏中的"科技"被重新定义为"知识"，使其涵盖面更广（包括技术发明、社会制度、学术知识等），以便更好地体现文明发展的全貌。

#### 验收标准

1. WHEN 玩家打开游戏 THEN 系统 SHALL 将所有原"科技"相关的 UI 文本显示为"知识"（包括 Tab 标题、提示文字、日志消息等）。
2. WHEN 内部代码引用原科技概念时 THEN 系统 SHALL 保持 `techsUnlocked`、`TECHS` 等内部变量名不变（避免大规模重构），仅在用户可见的展示层替换文案。
3. WHEN 资源 `science` 被展示时 THEN 系统 SHALL 将其显示名称改为"学识"（icon 和 color 不变）。

### 需求 1.2 — 知识树结构

**用户故事：** 作为一名玩家，我希望知识以树状结构呈现，必须先研发前置知识才能研发后续知识，以便我感受到文明发展的因果链和策略深度。

#### 验收标准

1. WHEN 加载知识配置时 THEN 系统 SHALL 为每个知识节点解析 `prerequisites` 字段（字符串数组），确定其前置依赖。
2. IF 某知识节点没有 `prerequisites` 字段 THEN 系统 SHALL 将其视为该时代的根节点（入口知识），仅受时代限制。
3. WHEN 玩家试图研究一个知识 AND 该知识的任一前置知识未被解锁 THEN 系统 SHALL 阻止研究并显示缺失的前置知识名称。
4. WHEN 玩家在知识树 UI 中浏览时 THEN 系统 SHALL 以图形化的树状/网络布局展示知识节点和它们之间的依赖连线，替代当前的平铺列表。
5. WHEN 知识 A 是知识 B 的前置 THEN UI SHALL 用有向连线（箭头）从 A 指向 B，并根据解锁状态着色：
   - 已研究的连线：明亮/高亮
   - 可研究的连线（前置已满足）：半透明
   - 锁定的连线（前置未满足）：暗淡/灰色
6. WHEN 每个时代的知识树被渲染时 THEN 系统 SHALL 按照依赖深度自动排列节点层级（根节点在左/上，末端节点在右/下），同一层的节点水平/垂直均匀分布。

### 需求 1.3 — 知识数据补全前置关系

**用户故事：** 作为一名游戏设计者，我希望所有现有知识节点都被赋予合理的前置依赖关系，以便形成有意义的研发路径。

#### 验收标准

1. WHEN 知识配置被加载时 THEN 系统 SHALL 验证每个 `prerequisites` 引用的 id 都存在于 TECHS 数组中，不存在的引用视为配置错误并在控制台警告。
2. WHEN 设计前置关系时 THEN 系统 SHALL 保证不出现循环依赖。
3. WHEN 补全前置关系时 THEN 设计 SHALL 遵循以下原则：
   - 同一时代内的知识应形成至少 2-3 条独立的发展路线（如经济线、军事线、文化线）
   - 跨时代的衔接点应自然合理（如"基础灌溉"→后续农业技术）
   - 已有 `prerequisites` 的军事科技线（gunpowder_formula → musket_manufacturing → rifle_engineering → military_industrialization）保持不变
   - 电气/原子/信息时代已有的 `prerequisites` 链条保持不变
4. IF 单个时代内的知识数量超过 8 个 THEN 设计 SHALL 保证树的最大宽度（同层并列节点数）不超过 6，以确保 UI 可读性。

### 需求 1.4 — 知识 Tab 改为多子 Tab 结构

**用户故事：** 作为一名玩家，我希望"知识" Tab 下包含"知识树"和"理念"两个子 Tab，以便在同一入口下管理所有知识/思想相关的功能。

#### 验收标准

1. WHEN 玩家点击主导航栏的"知识" Tab THEN 系统 SHALL 显示一个包含子 Tab 切换的界面，默认显示"知识树"子 Tab。
2. WHEN 玩家在子 Tab 栏切换到"理念"子 Tab THEN 系统 SHALL 切换到理念系统的界面。
3. WHEN 子 Tab 结构被实现时 THEN 系统 SHALL 参考现有军事 Tab / 政治 Tab 的分 Tab 实现模式（如有），保持 UI 风格一致。

## 第二部分：理念系统（全新 Ideology/Ethos 系统）

### 需求 2.1 — 理念分数与获取

**用户故事：** 作为一名玩家，我希望通过游戏中的具体行为和成就来积累"理念分数"，以便我感受到自己的游戏行为塑造了文明的思想特征。

#### 验收标准

1. WHEN 玩家完成特定游戏行为时 THEN 系统 SHALL 根据行为类型奖励对应数量的理念分数。触发行为包括但不限于：
   - 研发知识（每个知识奖励基础分数，高级知识更多）
   - 进入新时代（大量分数）
   - 达成成就（基于成就稀有度给予分数）
   - 建造特定数量的建筑（里程碑式触发）
   - 人口达到特定阈值
   - 完成特定政令
   - 战争胜利/失败
   - 产业链完整度达标
   - 贸易达到特定规模
   - 特定阶层满意度达到阈值
   - 文化/科研产出达到阈值
2. WHEN 理念分数累积达到当前阈值时 THEN 系统 SHALL 触发一次"理念涌现"事件。
3. WHEN 理念涌现被触发时 THEN 系统 SHALL 按以下公式计算下一次涌现的阈值：
   - 基础阈值 = `50 + 已拥有理念数 × 30`（递增，鼓励早期积极获取，后期更难）
   - 该公式的具体数值待平衡测试调整
4. WHEN 理念分数变化时 THEN UI SHALL 在"知识" Tab 的理念子 Tab 中实时显示当前分数和下次涌现阈值的进度条。

### 需求 2.2 — 理念涌现（三选一机制）

**用户故事：** 作为一名玩家，我希望每次理念涌现时能从 3 个随机理念中选择 1 个，以便感受到每局游戏的独特性和策略选择的乐趣。

#### 验收标准

1. WHEN 理念涌现被触发时 THEN 系统 SHALL 从当前可用理念池中随机抽取 3 个理念呈现给玩家。
2. WHEN 抽取理念时 THEN 系统 SHALL 考虑以下过滤条件：
   - 理念的时代要求（`unlockEpoch`）<= 当前时代
   - 理念的特殊解锁条件（若有）已满足
   - 权重机制：部分理念在特定条件下概率更高（如战争中更容易出现军事理念）
3. WHEN 3 个候选理念中出现玩家已拥有的理念时 THEN 系统 SHALL 正常显示该理念，但标记为"强化"选项。
4. WHEN 玩家选择一个已拥有的理念进行强化时 THEN 系统 SHALL 提升该理念的等级（最高 3 级），并增强其数值效果或解锁新的机制效果：
   - 等级 1→2：数值效果 ×1.5 或解锁额外条件效果
   - 等级 2→3：数值效果 ×2.0 或解锁终极联动效果
5. WHEN 3 个候选理念全部已达最高等级时 THEN 系统 SHALL 重新抽取直到至少有 1 个可选理念（含未拥有或未满级的）。
6. WHEN 理念涌现弹窗显示时 THEN UI SHALL 以卡牌形式展示 3 个候选理念，包含：
   - 理念名称、图标、所属类别
   - 效果描述（区分当前等级效果和下一级预览）
   - 如果是强化选项，显示当前等级和升级后变化
7. WHEN 玩家做出选择后 THEN 系统 SHALL 关闭涌现弹窗，将理念加入玩家的理念库（或升级已有理念）。

### 需求 2.3 — 理念卡槽系统

**用户故事：** 作为一名玩家，我希望将获得的理念装备到有限的卡槽中，以便我需要策略性地选择当前激活的理念组合。

#### 验收标准

1. WHEN 玩家获得理念时 THEN 系统 SHALL 将理念加入"理念库"（所有已获得的理念集合）。
2. WHEN 玩家查看理念界面时 THEN UI SHALL 显示：
   - 顶部：理念卡槽区域（当前已装备的理念卡牌）
   - 下方：理念库（所有已获得但未装备的理念卡牌）
3. WHEN 游戏开始时 THEN 系统 SHALL 提供 3 个基础卡槽。
4. WHEN 特定条件满足时 THEN 系统 SHALL 解锁额外卡槽：
   - 进入青铜时代：+1 卡槽（总计 4）
   - 进入古典时代：+1 卡槽（总计 5）
   - 进入封建时代：+1 卡槽（总计 6）
   - 进入探索时代：+1 卡槽（总计 7）
   - 进入启蒙时代：+1 卡槽（总计 8）
   - 特定知识研发或理念联动可额外解锁卡槽（最终上限 10）
5. WHEN 玩家尝试装备理念到已满的卡槽 THEN 系统 SHALL 提示玩家先移除一个已装备的理念。
6. WHEN 玩家将理念从卡槽中移除时 THEN 系统 SHALL 立即取消该理念的所有效果，并将其放回理念库。
7. WHEN 理念被装入/移出卡槽时 THEN 系统 SHALL 施加一个冷却期（如 30 个游戏日），期间该理念不能再被重新装入/移出（防止频繁切换）。

### 需求 2.4 — 理念效果体系

**用户故事：** 作为一名玩家，我希望理念能产生深刻且多样的效果，不仅是简单数值加成，还有类似《小丑牌》的条件触发和联动效果，以便我享受 build 构建的深度策略乐趣。

#### 验收标准

##### 2.4.1 基础数值效果

1. WHEN 理念被装备到卡槽时 THEN 系统 SHALL 应用该理念定义的基础数值效果，效果类型包括（复用现有 `effects` 体系）：
   - `buildings: { buildingId: value }` — 特定建筑产出加成
   - `categories: { category: value }` — 类别产出加成
   - `incomePercent` — 财政收入加成
   - `maxPop` — 人口上限加成
   - `scienceBonus` — 科研加成
   - `cultureBonus` — 文化加成
   - `militaryBonus` — 军事力量加成
   - `stability` — 稳定度修正
   - `taxIncome` — 税收修正
   - `production` — 全局生产效率
   - `perPopPassive: { resource: amount }` — 每人口被动产出

##### 2.4.2 条件触发效果（Trigger Effects）

2. WHEN 理念定义了 `triggerEffects` 字段时 THEN 系统 SHALL 在满足条件时动态计算额外效果。条件触发效果的模式包括：

   **a) 阶层关联型**
   - 示例："工人阶级额外提供 0.02 科研/人"
   - 数据结构：`{ type: 'stratum_bonus', stratum: 'worker', bonus: { perPopPassive: { science: 0.02 } } }`
   - WHEN 该理念激活时 THEN 系统 SHALL 为指定阶层的每个人口添加额外被动产出。

   **b) 人口比例型**
   - 示例："自耕农人口数量的 1% 被算为额外战斗力"
   - 数据结构：`{ type: 'pop_ratio_bonus', stratum: 'peasant', ratio: 0.01, target: 'militaryPower' }`
   - WHEN 计算军事力量时 THEN 系统 SHALL 将指定阶层人口 × 比例系数加到目标属性上。

   **c) 产业链关联型**
   - 示例："每拥有一个完整产业链，获得 10% 军事力量加成"
   - 数据结构：`{ type: 'chain_count_bonus', countType: 'complete', perCount: { militaryBonus: 0.10 } }`
   - WHEN 产业链完整度变化时 THEN 系统 SHALL 重新计算该效果的加成值。

   **d) 知识计数型**
   - 示例："每研发一个知识，获得 10 人口"
   - 数据结构：`{ type: 'tech_count_bonus', perTech: { flatPop: 10 } }`
   - WHEN 新知识被研发时 THEN 系统 SHALL 触发一次性效果（如增加人口）或持续效果（如按知识数量的加成）。

   **e) 资源阈值型**
   - 示例："当银币储备超过 10000 时，所有建筑产出 +5%"
   - 数据结构：`{ type: 'resource_threshold', resource: 'silver', threshold: 10000, bonus: { production: 0.05 } }`
   - WHEN 资源量变化时 THEN 系统 SHALL 检查阈值条件并动态启停该效果。

   **f) 建筑计数型**
   - 示例："每 10 座军事建筑提供额外 5% 采集加成"
   - 数据结构：`{ type: 'building_count_bonus', category: 'military', per: 10, bonus: { categories: { gather: 0.05 } } }`
   - WHEN 建筑数量变化时 THEN 系统 SHALL 重新计算效果。

   **g) 时代关联型**
   - 示例："在当前时代及之后，所有产出额外 +2% 每时代"
   - 数据结构：`{ type: 'epoch_scaling', perEpoch: { production: 0.02 } }`
   - WHEN 时代变化时 THEN 系统 SHALL 用当前时代编号计算累积效果。

##### 2.4.3 联动效果（Synergy Effects）

3. WHEN 多个特定理念同时被装备到卡槽时 THEN 系统 SHALL 检查是否触发联动效果组合：
   - 数据结构：定义在 `IDEOLOGY_SYNERGIES` 配置中
   - 每个联动包含：所需理念 id 列表、联动名称、联动效果
   - 示例联动：
     - "人文主义" + "社会契约" → "启蒙精神"：科研 +15%，文化 +15%，稳定度 +10
     - "军国主义" + "一神教" → "圣战"：军事 +25%，异教国家外交 -20
     - "重商主义" + "特许殖民" → "贸易帝国"：贸易收入 +30%，每商人人口额外 +0.05 银币
4. WHEN 联动效果激活时 THEN UI SHALL 在理念卡槽区域显示联动标识和效果说明。
5. WHEN 联动所需的任一理念被移出卡槽时 THEN 系统 SHALL 立即取消联动效果。

### 需求 2.5 — 理念分类与内容设计

**用户故事：** 作为一名玩家，我希望理念涵盖哲学、神学、政治、经济、美学等多个维度，且每个理念都有史实背景和独特的游戏效果，以便我感受到思想史的丰富性和策略深度。

#### 验收标准

1. WHEN 理念库被设计时 THEN 设计 SHALL 包含以下分类（每个分类至少 5-8 个理念，总量目标 60-80 个）：

   | 分类 | 说明 | 示例理念 |
   |------|------|----------|
   | 哲学理念 | 认识论、方法论、世界观 | 人文主义、理性主义、存在主义、唯物辩证法、实用主义、怀疑论、犬儒主义 |
   | 神学理念 | 宗教信仰、神学体系 | 一神教、多神教、祖先崇拜、泛神论、无神论、弥赛亚信仰、业报轮回 |
   | 政治理念 | 治国思想、政治主张 | 君权神授、共和主义、绝对君主制、民族主义、社会主义、自由主义、法西斯主义、无政府主义 |
   | 经济理念 | 经济学说、贸易思想 | 重商主义、自由放任、重农主义、国家资本主义、计划经济、福利国家 |
   | 军事理念 | 战争哲学、军事学说 | 军国主义、和平主义、全民皆兵、职业军人、游击战术、海权论 |
   | 美学理念 | 艺术潮流、文化运动 | 古典主义、浪漫主义、印象派、达达主义、极简主义、包豪斯 |
   | 科学理念 | 知识方法论、技术观 | 经验主义、科学方法、技术乐观主义、卢德主义 |
   | 社会理念 | 社会组织、阶级关系 | 种姓制度、平等主义、精英主义、女性主义、公民社会 |

2. WHEN 每个理念被设计时 THEN 设计 SHALL 确保：
   - 具有明确的 `unlockEpoch`（最早可出现的时代）
   - 具有 1-3 级的效果递进设计
   - 至少参与 1 个联动组合
   - 效果包含至少 1 个条件触发效果（非纯数值加成）
   - 具有简短的史实/文化背景描述

3. WHEN 玩家在理念界面浏览时 THEN UI SHALL 支持按分类筛选和查看所有理念。

### 需求 2.6 — 理念与现有系统的整合

**用户故事：** 作为一名玩家，我希望理念效果能真正渗透到游戏的各个系统中，而不是孤立的数值层，以便理念选择成为核心策略维度。

#### 验收标准

1. WHEN 理念效果被应用时 THEN 系统 SHALL 在现有的 `effects.js` 效果叠加管道中插入理念效果层，叠加顺序为：
   - 时代加成 → 科技加成 → 政令加成 → 政体加成 → **理念加成** → Buff/Debuff → 事件加成
2. WHEN 理念提供阶层相关效果时 THEN 系统 SHALL 影响：
   - 阶层的被动产出
   - 阶层的需求系数
   - 阶层的满意度修正
   - 阶层的影响力权重
3. WHEN 理念提供军事效果时 THEN 系统 SHALL 影响：
   - 军事力量计算
   - 招募速度/成本
   - 兵种能力修正
4. WHEN 理念提供经济效果时 THEN 系统 SHALL 影响：
   - 建筑产出计算
   - 市场价格修正
   - 贸易收益系数
   - 税收效率
5. WHEN 理念提供外交效果时 THEN 系统 SHALL 影响：
   - 外交关系修正
   - 条约效果修正
   - 贸易意愿
6. WHEN 理念效果影响游戏状态时 THEN 系统 SHALL 在相关 UI 面板的 tooltip 中显示"理念加成"来源（如建筑产出 tooltip 应列出理念贡献的部分）。

### 需求 2.7 — 理念分数的行为触发细则

**用户故事：** 作为一名玩家，我希望理念分数的获取方式多样且有意义，以便我的每一个重要决策都推动文明思想的演进。

#### 验收标准

1. WHEN 以下行为发生时 THEN 系统 SHALL 奖励对应的理念分数（参考值，待平衡）：

   | 行为类型 | 触发条件 | 基础分数 | 说明 |
   |----------|----------|----------|------|
   | 研发知识 | 每次成功研发 | 5 + epoch × 3 | 高时代知识给更多分 |
   | 进入新时代 | 每次时代升级 | 30 + epoch × 10 | 时代跃迁是思想跃迁 |
   | 成就达成 | 每次解锁成就 | 10-30（按稀有度） | 复用现有成就系统 |
   | 建筑里程碑 | 建筑总数达50/100/200... | 15 每里程碑 | 物质基础催生思想 |
   | 人口里程碑 | 人口达100/500/1000... | 20 每里程碑 | 人多思想多 |
   | 战争结果 | 胜利或失败 | 15-25 | 战争催生反思 |
   | 产业链完成 | 每条链首次完成 | 20 | 产业成熟催生理论 |
   | 贸易里程碑 | 贸易收入累计达标 | 10 每里程碑 | 交流催生思想 |
   | 阶层和谐 | 所有阶层满意度>70 | 15（首次） | 社会和谐催生哲学 |
   | 危机存活 | 稳定度曾跌到<20后回升 | 20 | 危机催生反思 |
   | 文化繁荣 | 文化总量达标 | 10 每里程碑 | 文化积淀催生理念 |

2. WHEN 理念分数被获得时 THEN UI SHALL 在游戏日志中显示获得的分数和原因。
3. WHEN 理念分数被获得时 THEN UI SHALL 在理念进度条处有短暂的增长动画。

### 需求 2.8 — 理念涌现的权重系统

**用户故事：** 作为一名玩家，我希望涌现的理念与我的游戏风格和当前局势相关联，以便理念系统感觉是有机的文明演化，而非纯粹随机。

#### 验收标准

1. WHEN 理念涌现被触发时 THEN 系统 SHALL 根据以下因素调整各理念的出现权重：
   - **时代匹配**：当前时代 = `unlockEpoch` 的理念权重 ×2
   - **行为关联**：最近 30 天内的主要行为（如大量建造军事建筑则军事理念权重 ×1.5）
   - **阶层关联**：执政联盟中的阶层相关理念权重提升
   - **稀有度**：部分理念设为稀有（权重基础值更低），但在特殊条件下权重大幅提升
   - **互斥/限制**：部分理念互斥（如"一神教"和"多神教"同时出现概率降低但不为零）
2. WHEN 权重计算完成后 THEN 系统 SHALL 使用加权随机算法从理念池中抽取 3 个。

### 需求 2.9 — 存档与持久化

**用户故事：** 作为一名玩家，我希望理念系统的状态（已获得的理念、等级、卡槽配置、理念分数）能被正确保存和加载。

#### 验收标准

1. WHEN 游戏保存时 THEN 系统 SHALL 将以下数据序列化到存档中：
   - 理念分数（当前值和已消耗总量）
   - 理念库（已获得的理念 id 列表及其等级）
   - 卡槽配置（已装备的理念 id 列表和槽位数量）
   - 理念冷却状态
   - 已触发的里程碑记录（防止重复奖励分数）
2. WHEN 游戏加载时 THEN 系统 SHALL 从存档中恢复所有理念状态，并重新计算所有激活理念的效果。
3. WHEN 存档版本不含理念数据时（向前兼容） THEN 系统 SHALL 初始化默认理念状态（空理念库、3 个基础卡槽、0 分数）。

### 需求 2.10 — UI/UX 设计规范

**用户故事：** 作为一名玩家，我希望理念系统的界面美观、直观、且与游戏整体风格一致。

#### 验收标准

1. WHEN 理念卡牌被渲染时 THEN UI SHALL 以卡牌样式展示，包含：
   - 卡牌背景：根据分类不同的颜色主题
   - 等级星标：1-3 星表示当前等级
   - 理念图标：使用 lucide-react 图标库
   - 名称和一句话描述
   - 效果列表（简洁的效果标签）
   - 联动提示（如果与卡槽中其他理念有联动）
2. WHEN 理念涌现弹窗显示时 THEN UI SHALL：
   - 全屏半透明遮罩
   - 3 张卡牌横向排列
   - 悬浮/点击查看详细效果
   - 选择动画（选中卡牌高亮放大，未选卡牌淡出）
   - 使用 framer-motion 动画
3. WHEN 理念卡槽界面显示时 THEN UI SHALL：
   - 卡槽呈水平排列
   - 支持拖拽或点击装备/卸下
   - 联动效果以光线/连线方式可视化
   - 空卡槽显示虚线轮廓
   - 锁定的卡槽显示锁图标和解锁条件

## 第三部分：技术约束与边界条件

### 需求 3.1 — 性能要求

**用户故事：** 作为一名玩家，我希望新系统不会影响游戏的流畅性。

#### 验收标准

1. WHEN 理念效果被计算时 THEN 系统 SHALL 在每个 simulation tick 中通过预计算缓存避免重复遍历：
   - 理念效果在装备/卸下时一次性计算并缓存
   - 条件触发效果在相关状态变化时才重新评估
   - 联动效果在卡槽变化时才重新评估
2. WHEN 知识树 UI 被渲染时 THEN 系统 SHALL 对不可见区域的节点进行虚拟化或延迟渲染。
3. IF 理念效果计算导致单 tick 时间超过 16ms THEN 系统 SHALL 将条件触发效果的评估分散到多个 tick 中。

### 需求 3.2 — 架构兼容性

**用户故事：** 作为一名开发者，我希望新系统尽量复用现有架构。

#### 验收标准

1. WHEN 理念效果需要被应用时 THEN 系统 SHALL 通过扩展现有的 `initializeBonuses` / `applyEffects` 函数来实现，而非创建并行效果管道。
2. WHEN 理念配置数据被存储时 THEN 系统 SHALL 使用与 `technologies.js` 类似的格式放在 `src/config/ideologies.js` 中。
3. WHEN 理念状态被管理时 THEN 系统 SHALL 扩展现有的 `useGameState.js` hook（或 zustand store），而非创建独立的状态管理系统。
4. WHEN 理念 UI 需要被创建时 THEN 系统 SHALL 在 `src/components/tabs/` 下创建理念相关组件，复用现有的 `UIComponents` 和样式模式。

### 需求 3.3 — 游戏平衡约束

**用户故事：** 作为一名游戏设计者，我希望理念系统不会破坏现有的经济和数值平衡。

#### 验收标准

1. WHEN 理念数值效果被设计时 THEN 设计 SHALL 遵循以下上限：
   - 单个理念的最大加成不超过同等级科技效果的 50%
   - 联动效果的最大加成不超过最强政体效果的 80%
   - 条件触发效果应有合理的上限（cap），防止极端情况下数值爆炸
2. WHEN 理念效果叠加时 THEN 系统 SHALL 对同类效果执行软上限：
   - 同类别（如全部军事加成）超过 100% 后进入递减区间
   - 具体递减公式：`实际加成 = 100% + (超出部分 × 0.5)`
3. WHEN 理念被设计时 THEN 设计 SHALL 确保不同 build 路线之间有明确的取舍（trade-off）：
   - 强化军事 → 牺牲经济或文化
   - 强化经济 → 牺牲军事或稳定度
   - 追求文化 → 牺牲生产效率
