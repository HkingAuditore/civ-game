# 地块系统（Tile System）策划案

> 适用项目：civ-game  
> 版本：v1.0（设计草案）  
> 定位：玩法设计蓝图。配套文档 `requirements.md`（工程验收清单）、`task-item.md`（任务清单）。

---

## 1. 设计总览

### 1.1 一句话定位
把"抽象聚合式国家经营"升级为"地理分片式文明建设"——玩家从"管理一个抽象国家"变成"经营一片有空间、有地理特征的国土"。

### 1.2 设计目标
| 编号 | 目标 | 验证方式 |
|---|---|---|
| G1 | 让玩家产生"地理意识"——意识到"哪里"比"多少"更重要 | 玩家会主动选平原建农田、山地建矿场 |
| G2 | 创造长线经营目标——通过扩张、占领、殖民逐步积累国土 | 游戏中后期玩家会为"拿下那块资源地"做长线规划 |
| G3 | 放大战争与外交的价值 | 战争胜利能直接带来"看得见的地皮"，而非仅一个数值奖励 |
| G4 | 不破坏现有经济的良好手感 | 市场/价格/贸易/税收体验与旧版接近，不让玩家陷入繁琐的调度 |
| G5 | 为未来扩展（地区叛乱、贸易路线、军队移动）留出结构基础 | 数据模型预留邻接表、归属翻转接口 |

### 1.3 非目标（本期不做）
- ❌ 每地块独立市场与跨地块运输成本
- ❌ 军队在地块间逐格推进、地形战术
- ❌ 地区性起义（阶层不满按地块触发）
- ❌ AI 的高智能地块经济决策（本期 AI 仅要求"会在地块上建造"）

### 1.4 核心设计原则
1. **空间感优先于精确性**——地形要能"一眼看出能干什么"，不追求地理学严谨
2. **共享市场隔离风险**——把空间性锁在"生产端"，不污染"流通端"
3. **老玩家零迁移成本**——老存档自动变成"单地块帝国"，手感基本不变
4. **UI 分层**——宏观看全国聚合、微观看地块细节，不强迫玩家时刻操作地图

## 2. 玩法闭环

### 2.1 核心循环（Core Loop）
```
选择地块 → 评估地形 & 资源加成 → 规划建筑布局 → 建造与雇佣本地人口 
    → 本地阶层积累财富 → 全国市场消化产出 → 税收回流国库
    → 国库支撑军队/外交 → 战争/外交获得新地块 → 回到起点
```

### 2.2 新手到精通的节奏设计
| 阶段 | 玩家心智 | 玩家操作重点 |
|---|---|---|
| **开局（1 地块）** | "这和旧版没区别啊？" | 只在首都建造，熟悉界面 |
| **扩张期（2–5 地块）** | "我得挑资源加成高的地块去占" | 观察地图，规划扩张方向 |
| **中期（5–15 地块）** | "每个地块都要差异化定位：这块做粮仓、那块做矿产" | 针对性布局，处理本地就业 |
| **后期（15+ 地块）** | "我要建一个能自洽的产业帝国" | 宏观布局，少数关键地块做优化 |

### 2.3 决策点（Decision Points）
每局游戏中玩家需要反复做的空间决策：

1. **"扩张哪块？"**——选中立地块？逼割他国地块？打下邻国？
2. **"新地块做什么？"**——住房地块（提供人口）/ 产业地块（提供产出）/ 军事地块（提供防御）
3. **"地块内建什么？"**——受承载上限限制，不能所有建筑都建
4. **"就业错配怎么办？"**——这块缺工（建太多厂）vs 那块失业（没岗位），玩家要用建筑布局来调节

## 3. 地图与地形设计

### 3.1 地图形态
- **形状**：方形网格（本期），单局规模 30–80 格
- **视角**：俯视 2D
- **大小分档**：
  - 小图：30 格（≈3-5 国，快速对局）
  - 中图：50 格（≈5-7 国，标准体验）
  - 大图：80 格（≈8-10 国，史诗体验）

### 3.2 地形表（10 种）

| 地形 | 中文 | 通行性 | 承载上限 | 主要产出加成 | 禁止产出 | 设计意图 |
|---|---|---|---|---|---|---|
| `plains` | 平原 | ✅ | 6 | food ×1.2, cotton ×1.1 | 无 | 最通用的开局地，粮仓首选 |
| `hills` | 丘陵 | ✅ | 4 | iron ×1.3, stone ×1.2 | food ×0.8 | 矿业与防御 |
| `mountain` | 山地 | ❌* | 2 | iron ×2.0, gold ×2.0, stone ×1.5 | food × 0, cotton × 0 | 稀有资源，但承载极低 |
| `river` | 河流 | ✅ | 5 | food ×1.3, fish ×1.5 | 无 | 粮食+水运（未来扩展贸易节点） |
| `ocean` | 海洋 | ❌* | 0 | fish ×2.0 | 几乎一切陆地产物 | 只允许特殊建筑（港口），未来扩展海运 |
| `desert` | 沙漠 | ✅ | 2 | gold ×1.5 | food ×0, cotton ×0.5 | 贫瘠地，但稀有资源点 |
| `plateau` | 高原 | ✅ | 3 | stone ×1.3, silver ×1.2 | food ×0.7 | 中庸地，混合加成 |
| `marsh` | 沼泽 | ✅ | 2 | 无 | food ×0.5, iron ×0.5 | 惩罚性地形，鼓励玩家改造或放弃 |
| `forest` | 森林 | ✅ | 4 | wood ×2.0, fur ×1.5 | food ×0.9 | 木材与皮草重镇 |
| `snow` | 雪原 | ✅ | 2 | fur ×1.3 | food ×0.5, cotton ×0 | 边疆地，承载低 |

*`passable: false` 不代表"什么都不能做"，而是"默认不能建造"——未来特殊建筑（港口/矿坑）可通过"豁免规则"在这些地块建造。

### 3.3 资源加成语义（关键）
- **加成值 = 1.0**：标准产出（默认）
- **加成值 = 0**：该资源完全无法在此地块产出（用于强制地理规则）
- **加成值 > 1.0**：产出乘以加成，如 2.5 表示 2.5 倍产出
- **加成值 < 1.0**：产出打折，如 0.5 表示产量减半

### 3.4 地图生成算法（设计层）
1. 用简化噪声函数划分大地形带（海洋、山脉、平原主导区）
2. 在主导区内按概率细化（平原区里散布丘陵、森林）
3. 河流从山脉向海洋延伸成链
4. 随机选一个 `plains` 或 `river` 作为玩家首都，**强制其相邻 2 格必须 `passable`**
5. 其他 nation 的首都按外交系统规则分散到地图其余位置
6. 剩余地块设为 `neutral`

## 4. 地块规则

### 4.1 建筑承载上限（buildingCapacity）
- 由地形决定，见 3.2 表
- 同种建筑叠加 1 格（即"3 座农田"占 3 格容量）
- 未来可通过"城市规划"类科技或建筑提升承载上限

### 4.2 归属（Owner）
| 值 | 含义 | 玩家可做 |
|---|---|---|
| `'player'` | 玩家拥有 | 建造、管理 |
| `'neutral'` | 无主之地 | 可殖民获得（未来迭代） |
| `'nation_xxx'` | 某外国拥有 | 可通过战争/外交获得 |

### 4.3 归属翻转时的数据迁移
当一块地的 owner 翻转时：
- 建筑、人口、阶层财富**随地块一起转移**——拿下肥地即得肥地的工业基础
- 本地阶层对"征服者"有满意度惩罚（未来可加"被征服 Debuff"）
- 本期版本先做最简版：建筑保留、人口保留、财富保留，无额外 Debuff

## 5. 建筑系统改造

### 5.1 建筑配置新增元数据
`buildings.js` 中每个建筑新增两个可选字段：

```js
{
  id: 'farm',
  // ...现有字段...
  allowedTerrain: ['plains', 'hills', 'river'],  // 允许建造的地形（白名单），不填视为无限制
  terrainBonus: { /* 可选，覆盖地块默认加成，通常不用 */ }
}
```

### 5.2 建造流程变化
旧流程：
```
BuildTab → buyBuilding(id) → buildings[id]++
```

新流程：
```
MapPanel 选中 tile → BuildTab 进入地块上下文 → buyBuilding(id, tileId) 
  → 校验 owner、terrain、capacity → tile.buildings[id]++
```

### 5.3 "特殊建筑"豁免通行性规则
某些建筑允许在"不可通行"地块上建造：
- **港口**：允许在 `ocean`、相邻 `ocean` 的 `coast` 上建造
- **矿井**：允许在 `mountain` 上建造（虽然 `mountain.passable: false`）

实现方式：建筑 `allowedTerrain` 字段优先级高于 `passable`。

### 5.4 建筑产出的地形加成计算
```
实际产出 = 基础产出 × 全局科技/理念加成 × 本地 resourceModifiers[resource]
```
产出计算后汇入**全国市场**（不按地块分市场）。

## 6. 人口与就业

### 6.1 本地就业原则
人口只能在本地块就业——这是本期最重要的玩法变化。

### 6.2 就业匹配流程（按地块独立执行）
```
for each player tile:
    for each stratum in tile.popStructure:
        本地岗位 = sum(tile.buildings[b].jobs[stratum])
        本地人口 = tile.popStructure[stratum]
        if 岗位 > 人口: 标记"本地缺工"
        if 岗位 < 人口: 标记"本地失业"
        按现有匹配规则填充岗位
```

### 6.3 地块失业处理
- **本期简化**：失业人口留在本地，不产生工资，仅消耗生活必需品，按现有规则可能下降阶层
- **未来扩展**：失业人口按吸引力自动迁移到邻接地块

### 6.4 人口增长规则
- 只有"住房建筑" (产出 `maxPop`) 能提供人口槽位
- 人口自然增长仍按现有模型，但"承载人口上限"变为**本地 maxPop**
- 地块间不自动迁移，本期不做"人口调度"UI

### 6.5 人口结构可视化
地图选中地块后的侧栏展示：
```
┌─ 平原 · 长安首都 ─────────┐
│ 人口：120 / 150          │
│  ├ 农民  80              │
│  ├ 工匠  25              │
│  └ 贵族   5              │
│ 就业率：92% (本地缺工 11) │
└─────────────────────────┘
```

## 7. 阶层财富与工资

### 7.1 地块化的记账
每个地块独立记录：
```js
tile.classFinancialData = {
  farmer: { wealth, income, expense },
  artisan: { ... },
  capitalist: { ... },
  ...
}
```

### 7.2 全国聚合供现有系统使用
- "阶层不满 → 革命"等现有子系统读取**全国聚合值**（各地块求和）
- UI 默认展示全国视图，可下钻到地块
- 这意味着本期"富人不满"依然是全国级事件，不做"地区起义"

### 7.3 为什么这样设计？
这是一个**渐进改造**的决定：
- 现在把数据存到地块 → 为未来"地区叛乱"留结构
- 现在聚合后再用 → 不破坏现有玩法手感
- 后续一个独立迭代，把"阶层不满"改为按地块触发

## 8. 市场、税收、贸易（保持全国级）

### 8.1 明确保持不变的子系统
| 子系统 | 当前实现 | 本期是否改动 |
|---|---|---|
| 资源池 `resources` | 全国池 | ❌ 不变 |
| 价格 `prices.js` | 全国供需 | ❌ 不变 |
| 贸易 `trading.js` | 全国级 | ❌ 不变 |
| 税收 `taxes.js` | 全国级 | ❌ 不变 |
| 理念 `ideology/*` | 全国级 | ❌ 不变 |
| 外交 `diplomacy/*` | 全国级 | ❌ 不变 |

### 8.2 设计理由
这些系统已有丰富的玩法内容和平衡。把它们全部下沉到地块意味着重写 40%+ 代码，风险爆炸。把空间性锁在"生产端"（建筑/人口/就业）已经能带来 80% 的玩法提升。

## 9. 战争与外交

### 9.1 外交"割地"改造
- **旧机制**：调整 `maxPop` / `population` 标量，无"地皮"概念
- **新机制**：翻转一个或多个具体地块的 `owner`
- **UI 变化**：谈判界面允许玩家选择"要哪些地块"，而不是只给一个总数字

### 9.2 战争胜利结算
战争结束时，战胜方可以：
1. 强迫战败方割让 N 个**相邻于战胜方国土**的地块
2. N 的上限由战争规模/胜利程度决定
3. 战败方保留首都（防止一战灭国）

### 9.3 占领地块的文化冲突（未来扩展，本期不做）
- 占领他族地块会有"被征服"不满 Debuff
- 需经过一段时间同化

## 10. AI 行为改造

### 10.1 AI 必须做的最小改动
AI 在执行"建造"决策时，必须指定一个自己拥有的地块：
```js
// 旧代码
ai.build(buildingId)
// 新代码
ai.build(buildingId, tileId)
```

### 10.2 AI 地块选择策略（由简到繁）
**V1（最简，本期默认）**：
- 从 AI 拥有的地块中筛选"未满 & 地形允许"
- 随机选一个

**V2（可选本期或下期）**：
- 按"地形加成值"加权选择（如建农田优先选 `plains`）
- 跳过承载已满的地块

**V3（未来）**：
- 按 AI 人格特质（如"工业狂魔"偏好建工厂）选择
- 按全国产业缺口选择

### 10.3 AI 扩张（未来）
- 本期 AI 不主动扩张新地块，只在已有地块上建造
- 未来迭代加入"AI 也会发动战争要地块"

## 11. UI 设计

### 11.1 核心 UI 层级变化

本期对底部导航做**结构性调整**：

- **旧结构**：底部导航 `[资源][建筑][人口][外交]...`，点击“建筑”直接进入建筑列表（全国笼统建造）
- **新结构**：底部导航 `[资源][🗺️地图][人口][外交]...`，**用“地图”取代“建筑”入口**
  - 一级界面（点“地图”）= 俯视地图 + 地块选择
  - 二级界面（点地块）= 该地块的建造面板与详情

**设计理由**：
1. 玩家心智必须强制从“笼统建造”切换到“空间建造”——不保留旧入口，避免玩家习惯性绕开地图
2. 建造永远绑定具体地块——从 UI 层面彻底消除“没有 tileId 的建造”这种模糊状态
3. 简化底部导航——不增加按钮数量，不挤压其他 tab

### 11.2 一级界面：地图视图（取代原 BuildTab 入口）


- 展示**可建造建筑列表**（复用原 BuildTab 卡片组件，加禁用原因提示）
- 建造按钮作用于**当前地块**
- 左上角“← 返回地图”回到一级界面

### 11.4 非我方地块的二级界面（查看模式）

点击中立/敌国地块时，进入**只读二级界面**：
- 展示地块信息、归属国家
- 不展示建造列表
- 提供相关外交入口：`[派遣使节]` `[宣战]` `[殖民]`（功能入口复用现有外交模块）

### 11.5 原建筑入口（BuildTab）的命运

- 底部导航的“建筑”按钮被替换为“地图”
- 原 BuildTab 的**卡片组件**被复用于二级界面的“可建造”列表
- 原 BuildTab 的**筛选/收藏/搜索**功能迁移到二级界面内部
- **不再保留“全国视图默认进建造”**——所有建造必须通过地图 → 地块路径

**例外：快捷入口（可选，本期不做）**
- 未来可在全国概览仪表盘提供“快速建造”按钮，自动选择首都作为 tileId
- 本期为了让玩家彻底习惯地块化玩法，刻意不提供捷径

## 12. 数值骨架与平衡

### 12.1 地块承载上限分布
| 档位 | 上限 | 适用地形 | 设计意图 |
|---|---|---|---|
| 极小 | 2 | 山地、沙漠、沼泽、雪原 | 稀有/贫瘠地，迫使玩家"好钢用在刀刃上" |
| 小 | 3 | 高原 | 中等地 |
| 中 | 4 | 丘陵、森林 | 主力产业地 |
| 大 | 5–6 | 平原、河流 | 首都/核心经济地 |

### 12.2 玩家开局资源（推荐）
- 首都 1 格（平原），承载上限 6
- 相邻 2 格保证可通行（rnd 地形：大概率 plains/forest/hills）
- 玩家初始人口与旧版相同（5），全部分配在首都
- 初始建筑与旧版相同

### 12.3 扩张节奏（预期）
| 时间点 | 预期地块数 | 获取方式 |
|---|---|---|
| 开局 | 1 | 初始 |
| ~年 10 | 3–5 | 殖民中立地（未来扩展）+ 少量外交 |
| ~年 30 | 8–12 | 外交割地 + 小规模战争 |
| ~年 50 | 15+ | 大战争 |

### 12.4 地形加成系数平衡
- 最强加成 ×2.5 左右（超过 ×3 会让玩家压力过大去争夺）
- 惩罚系数最低 0.5（不至于毫无产出）
- 绝对禁止（×0）仅用于"概念上合理"的组合（山地无粮食）

### 12.5 就业失配的惩罚强度
本期设定：
- 本地失业率 > 30%：阶层每月财富 -5%（可能下降阶层）
- 本地缺工率 > 30%：建筑产出 ×0.8（按岗位填充率线性递减）

这两项数值需要实装后调优。

## 13. 风险与权衡

### 13.1 已识别的风险

| 风险 | 严重度 | 缓解 |
|---|---|---|
| 本地就业导致"失业+缺工并存"玩家体感差 | 🔴 高 | 提供 UI 清晰告警；后续扩展自动迁移 |
| 阶层财富聚合后削弱"富人不满"事件强度 | 🟡 中 | 单独迭代"地区起义" |
| 存档兼容老版本的数据迁移出错 | 🔴 高 | 保留聚合冗余字段；加版本号；写迁移单元测试 |
| 地图 UI 在低端移动设备卡顿 | 🟡 中 | 限制地图规模；用 CSS Grid 而非 Canvas |
| AI 按地块决策导致 AI 变弱 | 🟢 低 | V1 用"随机已拥地块"保底 |
| 工期失控 | 🟡 中 | 分阶段上线 + 功能开关可回滚 |

### 13.2 关键权衡

1. **"本地就业"vs"全国调配"**：选了前者，换来空间感、牺牲一部分便利
2. **"阶层财富按地块存储但聚合后使用"vs"直接按地块使用"**：选了前者，保留未来扩展性、不破坏现有事件
3. **"全国市场"vs"地块市场"**：选了前者，保持贸易/价格系统复杂度可控
4. **"10 种地形"vs"5 种地形"**：选了前者，地理多样性更丰富、但需要更多地形图配表
