# 地块系统（Tile System）需求文档

## 引言

civ-game 当前采用"全局聚合模拟器"模型：所有建筑、人口、资源在全国层面以标量/字典聚合存储（`buildings: {id: count}`、`popStructure: {stratum: count}`、`resources: {id: amount}`），不存在"地理位置"概念。玩家在 BuildTab 中笼统地"建造 3 座农田"，而不是"在某地建造农田"。

本需求的目标是为游戏引入**地块系统**：一张类似文明系列的随机生成地图，由若干地块（tile）组成，每个地块拥有独立的地形、资源加成、建筑、人口、阶层财富和归属。玩家选中地块后进行针对性建设、就业与管理，而非笼统操作全国。

### 架构决策（关键 · 方案 A+：中型分片 + 全局市场）
经过现状评估与用户追加设计要点（本地就业 / 阶层财富独立），本需求采用**中型分片方案（A+）**：

| 模块 | 是否分片到地块 | 说明 |
|---|---|---|
| 建筑 `buildings` | ✅ 是 | `tile.buildings = {id: count}` |
| 人口 `popStructure` | ✅ 是 | `tile.popStructure`，并派生全国聚合视图 |
| 就业 / 岗位匹配 | ✅ 是 | **人口只能在本地块就业**，每地块独立匹配 |
| 工资 | ✅ 是 | 基于本地块岗位与本地阶层计算 |
| 阶层财富 `classFinancialData` | ✅ 是 | 每地块独立统计 |
| 市场 / 价格 | ❌ 否 | **全国共享一个市场**，全部地块产出汇入公共市场由供需定价 |
| 税收 / 贸易 / 理念 / 外交 / 稳定 | ❌ 否 | 全国级，无需改动 |
| 建筑配置 `buildings.js` | ❌ 否 | 配置本身不变，仅新增"允许地形"元数据 |

这意味着：
- **市场/定价/贸易/税收/理念/外交系统保持不变**，风险被隔离在人口-就业-工资链路内；
- 主模拟循环在"产出阶段"按地块遍历后合并到全局市场；
- "就业匹配"与"工资发放"需改写为按地块执行的版本；
- UI 在现有 BuildTab 之外新增地图面板，地块被选中后 BuildTab 进入"地块上下文模式"。

**不纳入本需求的范围**（留待未来迭代）：
- 每地块独立市场 / 跨地块运输成本；
- 地块间军事单位移动 / 战场推进；
- 地块级 AI 经济决策的复杂目标函数（本需求只要求 AI "可以在地块上建设"，不要求高智能）。

### 兼容性原则
- **老存档必须可加载**：自动生成单一"首都地块"，将全部既有建筑/人口/资源归属其中，游戏表现基本等价。
- **经济数值偏移可控**：升级后同一套建筑/人口配置，在地形加成 = 1 的情况下应产出与旧版相近的全国级产出/消费数值。

## 需求

### 需求 1：随机地图生成

**用户故事：** 作为一名玩家，我希望每局开始时拥有一张随机生成的类似文明系列的国土地图，以便每次游戏都有不同的地理挑战与资源分布。

#### 验收标准
1. WHEN 新开始一局游戏 THEN 系统 SHALL 根据随机种子生成一张由若干地块组成的地图（地图总地块数 30–80，玩家初始拥有地块数默认 1 个"首都"）。
2. WHEN 生成地图 THEN 系统 SHALL 基于噪声函数（如柏林噪声或简化版）分配地形，保证地形分布有地理合理性（如海洋成片、山脉连绵、平原主导）。
3. WHEN 生成地图 THEN 系统 SHALL 为每个地块计算并存储"邻接表"（与之相邻的其他地块 ID 列表），用于未来的扩张、战争、贸易路径计算。
4. WHEN 生成地图 THEN 中立地块与其他 nation 的地块 SHALL 按设定比例分配，其他 nation 的地块归属与 `config/countries.js` 中的现有外交对象对齐。
5. IF 随机生成结果导致玩家首都被海洋/山地包围无法扩张 THEN 系统 SHALL 在生成阶段重试或强制保证首都周围至少 2 个可通行的相邻地块。

### 需求 2：地块数据模型与地形

**用户故事：** 作为一名玩家，我希望每个地块有清晰的地形与资源倾向，以便我能根据地理优势决定在哪里建造什么。

#### 验收标准
1. WHEN 定义地形 THEN 系统 SHALL 提供 10 种地形：`plains`（平原）、`hills`（丘陵）、`mountain`（山地）、`river`（河流）、`ocean`（海洋）、`desert`（沙漠）、`plateau`（高原）、`marsh`（沼泽）、`forest`（森林）、`snow`（雪原）。
2. WHEN 定义地块 THEN 每个地块 SHALL 至少拥有以下字段：
   - `id`（地块 ID）
   - `adjacency`（邻接地块 ID 数组）
   - `terrain`（地形枚举）
   - `owner`（归属：`'player'` / `'neutral'` / 某个 nation id）
   - `resourceModifiers`（资源产出加成字典，如 `{ food: 1.0, iron: 0, cotton: 2.5 }`，未列出的资源视为 1）
   - `buildingCapacity`（建筑承载上限，数值）
   - `passable`（通行性，布尔）
   - `buildings`（本地块建筑字典 `{id: count}`）
   - `popStructure`（本地块人口结构 `{stratum: count}`）
   - `classFinancialData`（本地块阶层财富数据）
   - `discovered`（是否已探明，布尔）
3. WHEN 某建筑的产出资源在本地块 `resourceModifiers` 中值为 0 THEN 该建筑 SHALL 完全无法产出该资源（用于强制"山地无粮食"、"平原无铁矿"等地形规则）。
4. WHEN 某建筑产出资源在本地块加成为 2.5 THEN 该建筑在本地块上实际产出 SHALL 为标准产出 × 2.5。
5. WHEN 地块 `passable` 为 `false`（如海洋、高山） THEN 系统 SHALL 禁止在该地块上建造建筑或分配人口，仅允许特定类型（如港口，仅沿海/河流地块）通过其他规则豁免。
6. WHEN 加载老存档 THEN 系统 SHALL 自动生成 1 个"首都"地块（地形 `plains`、`owner: 'player'`、资源加成全部为 1、建筑承载上限足够大），把旧存档的 `buildings`、`popStructure`、`classFinancialData` 整体归入其中。

### 需求 3：建筑建造地块化

**用户故事：** 作为一名玩家，我希望只能在具体地块上建造建筑，而不是笼统地给全国"+1 农田"，以便感受经营的空间性。

#### 验收标准
1. WHEN 玩家点击建造按钮 THEN 系统 SHALL 将该建筑计入当前选中地块的 `tile.buildings`，而非全局 `buildings`。
2. WHEN 没有选中地块时调用建造 THEN 系统 SHALL 默认落到"首都"地块（兼容保护）。
3. WHEN 地块 `owner !== 'player'` THEN 系统 SHALL 禁止建造。
4. WHEN 地块地形与建筑所需地形不符（如山地不允许农田） THEN UI SHALL 禁用该建筑的建造按钮并显示禁用原因。
5. WHEN 地块上所有建筑总数已达到 `buildingCapacity` THEN 系统 SHALL 禁止继续建造并在 UI 显示"该地块已满"提示。
6. WHEN 建筑产出受地形加成 THEN 主模拟循环计算产出时 SHALL 把基础产出逐资源乘以本地 `resourceModifiers`，再汇入全局市场。
7. WHEN 不同地块的同一种建筑数量聚合 THEN 全局 `buildings[id]` SHALL 等于所有玩家地块内该建筑之和（兼容现有 UI）。

---

### 需求 4：人口与本地就业

**用户故事：** 作为一名玩家，我希望每个地块有自己的人口，并且人口只能在本地块就业，以便不同地块形成不同的人口/阶层结构与经济特色。

#### 验收标准
1. WHEN 地块上建造住房类建筑（产出 `maxPop`） THEN 本地块 SHALL 开始产生人口，上限 = 本地住房建筑提供的 `maxPop` 之和。
2. WHEN 地块上没有住房类建筑 THEN 本地块 `popStructure` 人口总数 SHALL 保持为 0。
3. WHEN 地块上新增工作类建筑 THEN 该建筑的岗位 `jobs` SHALL 只对本地块内的人口开放，不能由其他地块人口填充。
4. WHEN 某地块岗位供大于求 THEN 系统 SHALL 显示"本地用工短缺"并按现有就业规则填充本地阶层，不跨地块调人。
5. WHEN 某地块人口供大于求（失业） THEN 本地多余人口 SHALL 按现有失业规则处理（如下降阶层、迁出 —— 迁出目标可暂缓实现，本期仅保留"就地失业"）。
6. WHEN 主模拟循环执行"就业匹配"阶段 THEN 系统 SHALL 按地块逐个运行现有匹配算法，而非一次性在全国池内匹配。
7. WHEN 全局 `popStructure` 被旧 UI 读取 THEN 它 SHALL 等于所有玩家地块 `popStructure` 的逐阶层求和。

---

### 需求 5：阶层财富与工资地块化

**用户故事：** 作为一名玩家，我希望每个地块的阶层财富独立计算，以便不同地块可以出现不同的贫富差距与阶层冲突。

#### 验收标准
1. WHEN 地块上的岗位产出工资 THEN 工资 SHALL 进入本地块对应阶层的 `classFinancialData`，不汇入全国池再分配。
2. WHEN 本地块某阶层消费生活必需品 THEN 消费记账 SHALL 记在本地块 `classFinancialData` 上。
3. WHEN 全国级 UI 显示阶层财富 THEN 它 SHALL 由所有地块该阶层财富聚合而来（显示模式：全国求和 / 全国平均 / 按地块细分，具体由 UI 层决定）。
4. WHEN 阶层财富被用于触发满意度、需求、起义等全国级机制 THEN 系统 SHALL 采用"聚合后取全国值"的方式，保证这些子系统当前可以继续工作。
5. IF 后续迭代需要"阶层不满按地块触发地区性起义" THEN 本期 SHALL 保留阶层财富按地块存储的结构基础，但当期功能暂不实现。

---

### 需求 6：市场与税收保持全国级

**用户故事：** 作为一名玩家，我希望所有地块共用一个大市场，以便不必手动管理地块间的资源调运，游戏节奏不会被割裂。

#### 验收标准
1. WHEN 任何地块产出资源 THEN 资源 SHALL 直接汇入全国 `resources` 池，并由 `prices.js` 按全国供需定价。
2. WHEN 任何地块消费资源 THEN 资源 SHALL 从全国 `resources` 池扣除。
3. WHEN 税收计算 THEN 系统 SHALL 在每地块阶层财富上取应税部分，汇总入国库，规则保持与旧版一致。
4. WHEN 对外贸易执行 THEN `trading.js` SHALL 不感知地块，保持全国级行为。

---

### 需求 7：归属、战争与外交

**用户故事：** 作为一名玩家，我希望通过外交割地或战争胜利占领他国地块，以便形成长线国土经营目标。

#### 验收标准
1. WHEN 游戏开始 THEN 玩家 SHALL 拥有 1 个"首都"地块，其余地块为中立或隶属现有外交系统中的其他 nation。
2. WHEN 玩家通过外交"割让"获得土地 THEN 系统 SHALL 把对应 nation 下 1 个或多个地块的 `owner` 改为 `player`，地块上的人口与建筑一并转入玩家麾下。
3. WHEN 战争结束且胜利一方选择占领 THEN 胜利方 SHALL 能选择战败方的若干可占领地块，将其 `owner` 翻转到胜利方。
4. IF 地块 `owner !== 'player'` THEN 玩家 SHALL 无法在其上建造任何建筑。
5. WHEN 玩家失去地块（割让 / 战败 / 起义） THEN 该地块的建筑、人口、阶层财富 SHALL 从玩家聚合统计中剔除。
6. WHEN 现有外交"割地"机制被调用 THEN 它 SHALL 从"调整 `maxPop`/`population` 标量"改为"翻转一个或多个地块的 `owner`"，外部调用方签名保持兼容（内部实现变化）。

---

### 需求 8：AI 行为地块化

**用户故事：** 作为一名玩家，我希望 AI 对手也在地块上进行建设，以便游戏世界一致、可观察、可对抗。

#### 验收标准
1. WHEN AI 决策建造建筑 THEN AI 的"建造"动作 SHALL 作用于 AI 自己的某个具体地块，而非直接修改全局 `buildings`。
2. WHEN AI 选择建造地块 THEN 决策策略 SHALL 至少能做到"优先选择该建筑最匹配的地形（按资源加成）"与"跳过已满地块"。
3. WHEN AI 地块被玩家占领 THEN 该地块自动转入玩家名下，其上建筑与人口按"需求 7.5"处理。
4. IF 当前 AI 模块逻辑无法直接支持按地块决策 THEN 本期 SHALL 提供最小可行版本（AI 把所有建造随机分布在自己拥有的地块上），后续迭代再优化策略。

### 需求 9：UI 层级重构（建筑入口改为地图入口）

**用户故事：** 作为一名玩家，我希望点击底部"地图"按钮进入俯视地图，再点地块进入二级界面建造，而不是笼统地从"建筑"按钮直接建造，以便所有建造行为都明确绑定到具体土地。

#### 验收标准
1. WHEN 底部导航栏渲染 THEN 原"建筑"按钮 SHALL 被替换为"地图"按钮（图标 🗺️，文字"地图"），不保留独立的"建筑"一级入口。
2. WHEN 玩家点击"地图"按钮 THEN UI SHALL 进入一级地图界面，展示 2D 网格地图（本期首选方形），显示地形底色、归属边框颜色、首都标识、建筑数量/承载度徽标。
3. WHEN 玩家在一级地图点击自己拥有的地块 THEN UI SHALL 进入二级"地块建造与详情界面"，该界面以全屏接管面板或大型抽屉呈现（推荐全屏接管）。
4. WHEN 玩家在一级地图点击非自己拥有的地块（中立 / 他国） THEN UI SHALL 进入二级"地块查看界面"：只读信息 + 提供外交/宣战/殖民入口（功能入口复用现有外交模块），不显示建造列表。
5. WHEN 二级界面渲染 THEN 它 SHALL 至少包含以下分区：
   - 地块概览（地形、归属、资源加成、承载 used/max、人口/就业率）
   - 本地已有建筑列表（可拆除）
   - 可建造建筑列表（复用原 BuildTab 的卡片组件与筛选/搜索/收藏功能）
   - 顶部"← 返回地图"按钮
6. WHEN 二级界面的建造按钮被点击 THEN 建造动作 SHALL 作用于当前二级界面所对应的地块。
7. WHEN 玩家从二级界面点击底部其他 tab（如"资源"） THEN UI SHALL 直接切走，下次点击"地图"按钮 SHALL 回到一级地图（不记忆上次选中的地块）。
8. WHEN 玩家按 Android 返回键或 ESC THEN 二级界面 SHALL 退回一级地图。
9. WHEN 任何现有 UI 组件（模态框、图表、仪表板）读取 `buildings`、`popStructure`、`classFinancialData` THEN 系统 SHALL 提供"全局聚合视图"，保证这些组件无需改动即可继续运行。
10. WHEN 屏幕宽度小于移动断点 THEN 地图 UI SHALL 以可滚动/可缩放容器呈现，二级界面 SHALL 占满可视区。
11. WHEN 原 BuildTab 的筛选、搜索、收藏、卡片排序等已有功能被迁移 THEN 这些功能 SHALL 在二级界面内继续可用，不发生能力退化。
12. WHEN 本期不提供"快捷建造"入口 THEN 玩家 SHALL 无法绕过地图直接建造，以便强制建立"建造=地块行为"的玩法心智。

### 需求 10：存档兼容与分阶段上线

**用户故事：** 作为一名老玩家与开发者，我希望这次重构向后兼容且能分阶段验收，以便降低大改造风险。

#### 验收标准
1. WHEN 加载不含 `tiles` 的老存档 THEN 系统 SHALL 自动迁移为"单一首都地块"结构，游戏表现与升级前一致（资源加成全 1、本地就业匹配效果在单地块情况下等价于全国匹配）。
2. WHEN 保存新存档 THEN 存档 SHALL 包含 `tiles` 字段，并保留 `buildings`、`popStructure`、`classFinancialData` 的聚合冗余字段用于旧 UI / 调试 / 回滚。
3. IF 存档版本号低于"tile-system 版本号" THEN 加载时 SHALL 日志输出一次性迁移提示。
4. WHEN 第一阶段交付 THEN 系统 SHALL 提供完整 tiles 数据层 + 迁移，但主循环在"功能开关关闭时"与旧版 100% 行为等价。
5. WHEN 第二阶段交付 THEN 系统 SHALL 启用地块视图 UI、地块化建造、本地就业、阶层财富分片，允许内测。
6. WHEN 任一阶段发现重大回归 THEN 关闭功能开关 SHALL 能退回旧聚合行为。
7. WHEN 代码通过 `npm run lint` 与 `npm run build` THEN SHALL 无阻塞性错误。

### 需求 11：性能预算

**用户故事：** 作为一名玩家，我希望地块系统上线后游戏帧率不明显下降，以便保持流畅体验。

#### 验收标准
1. WHEN 玩家拥有 50 个地块执行主模拟 tick THEN tick 耗时 SHALL 不超过改造前平均值的 200%（本期上调容忍度，后续再优化）。
2. WHEN 某地块建筑数为 0 THEN 模拟循环 SHALL 跳过该地块的建筑/就业遍历。
3. WHEN 使用 `simulation.worker.js` THEN 地块数据 SHALL 能正确序列化传递给 worker，并在 worker 端完成按地块产出/就业计算。

## 风险与待确认事项（不作为验收标准，用于决策对齐）

1. **"阶层财富按地块独立计算"对现有机制的连锁影响**：
   - 现有"阶层不满 → 革命"、"理念触发阈值"、"外交情绪"等子系统读取的是全国级阶层财富。本需求约定"按地块存储、聚合后供全国子系统使用"，这会让"阶层不满"变得更平均，可能削弱现有"富人阶层不满"等戏剧性事件。请确认接受。
2. **"人口只能本地就业"对经济效率的影响**：
   - 地块内岗位与人口不匹配会出现"这边失业 + 那边缺工"。本期不做跨地块迁移，会显著改变玩家操作重点（从全国调配变成布局规划）。请确认这是预期的。
3. **工期提升**：
   - 相比最初的"方案 A"（2–3 周），本方案 A+ 因涉及 `jobs.js` + `wages.js` + `classFinancialData` 重写，预计工期 5–8 周（单人全职口径）。
4. **地图形状**：本需求默认方形网格。如需六边形请说明。
5. **初始地图规模与玩家初始领土**：默认 30–80 地块、玩家 1 个首都。如希望开局拥有 3–5 地块请说明。
