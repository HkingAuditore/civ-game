## 完成：AI 国家补给率始终 0% Bug 修复

### 根本原因
AI 国家的 `military.stockpile` 只补充 3 种资源（food、silver、materielKey），但军队 `maintenanceCost` 需要 iron、copper、cloth、stone、plank 等多种次要资源。`buildResourceCoverage` 取所有需求资源的最低覆盖率（`Math.min`），缺失任一种次要资源即补给率归零。

### 修复内容
1. **FrontViewPanel.jsx** — 战力面板补给率 `|| 1` → `?? 1`，修复 0 被 falsy 判定导致显示 100% 的问题
2. **aiWar.js `ensureAIMilitaryState`** — stockpile 初始化扩展为包含所有军队维护所需资源类型（wood、iron、copper、stone、cloth、plank、gunpowder、ammunition、steel、coal），根据时代和人口给予合理初始值
3. **aiWar.js `syncAINationMilitary`** — 每日补充逻辑改为根据军队实际 `calculateArmyMaintenance` 结果来补充各类次要资源（每日 1.2 倍日维护量）
4. **frontSystem.js `buildResourceCoverage`** — 从"取最低值"改为"加权分层计算"：核心资源（food/silver）占 70% 权重取最低值，次要资源占 30% 权重按需求量加权平均，避免微量资源缺失直接拉到 0%

## 修复：兵团 in_combat 状态死循环 + 幽灵引用导致未部署兵团参战

### 问题表现
1. 未部署到前线的兵团显示"战斗中"状态
2. 部署到战线后下一个 tick 兵团消失

### 根本原因
1. **in_combat 状态死循环**：`isInCombat = activeBattleCorpsSet.has(corps.id) || corps.status === 'in_combat'` 中 `|| corps.status === 'in_combat'` 导致一旦兵团被标记为 in_combat，即使战斗已结束也不会恢复
2. **状态恢复不完整**：恢复逻辑只处理 `deployed` 状态的孤立兵团，不处理 `in_combat` 状态的孤立兵团
3. **front.assignedCorps 幽灵引用**：兵团的 assignedFrontId 被恢复为 null 后，front.assignedCorps 中仍保留该 corps id

### 修复
1. `isInCombat` 判断只依赖 `activeBattleCorpsSet`
2. 增加 `isOrphanCombat` 检测恢复无 front 无活跃会战的 in_combat 兵团
3. 恢复后同步清理 `front.assignedCorps` 中的幽灵引用
4. 合并去重 `activeBattleCorpsSet`

- `src/hooks/useGameLoop.js`
